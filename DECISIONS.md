See `harness.yaml` for the machine-readable operating config (workflow gates, standard rules, epics/tasks, memory/checkpoint discipline) — `locked_decisions_ref` there points back at this file; don't duplicate locked decisions into `harness.yaml`, only reference them.

## Decided (do not silently reverse)
- 2026-07-19: State management = `flutter_bloc` (Bloc/Cubit), not GetX. Rationale: this fork already has 15+ global blocs wired in `main.dart` (`MultiBlocProvider`) plus feature-local blocs/cubits in every existing feature folder. Migrating to GetX for consistency with other projects (Onkur, FlexWork, Shell Agami) would require rewriting all working state logic app-wide for zero functional benefit and real regression risk. Confirmed by Dipu 2026-07-19.
- 2026-07-19: Prayer times calculated offline via `adhan_dart`, no API dependency.
- 2026-07-19: Repo initialized with git (previously untracked) and a baseline commit taken before any migration work, so Epic 0 dependency upgrades are reversible.
- 2026-07-19: `flutter_native_timezone` replaced with `flutter_timezone` (Dipu's call). Rationale: the old package's Kotlin code referenced `PluginRegistry.Registrar`/`.messenger()` — Flutter v1-embedding APIs removed entirely from the current engine. Not a config issue; the plugin itself was dead. One call site (`notification_service.dart`), drop-in swap (`FlutterTimezone.getLocalTimezone().identifier`).
- 2026-07-19: `motion_sensors` replaced with `flutter_qiblah` (Dipu's call, chosen over the initially-suggested `sensors_plus` after researching options). Rationale: same dead-API problem as flutter_native_timezone (`PluginRegistry.Registrar`/`.context`/`.messenger()`). `flutter_qiblah` is purpose-built and actively maintained, and handles location + sensor fusion + bearing math internally, so it replaced not just the sensor call but the app's own haversine bearing math and manual magnetometer smoothing too. This collapsed `QiblaBloc` + `AngleBloc` into a single `QiblaBloc` (via `emit.forEach(FlutterQiblah.qiblahStream)`) and deleted `angle_bloc/` entirely — effectively front-loading most of TASK-007 (Qibla migration) as a side effect of fixing the build. The folder move into `data/domain/presentation` for Qibla is still open. Known regression: the old "no magnetometer sensor" fallback UI (lottie + message) was dropped; `flutter_qiblah` has its own `androidDeviceSensorSupport()` + maps-fallback pattern that should replace it — follow-up for whenever TASK-007 is formally closed out.

- 2026-07-19: Dropped the per-step confirmation gate from the operating workflow (Dipu's explicit instruction). Previously (`AMAR_DEEN_PLAN.md` §6.1 / `harness.yaml` `workflow`), every task had to stop after "propose" and wait for a go-ahead before "implement". Now the workflow is analyze → propose → implement → report, with no stop-and-wait step — propose is still stated, just not gated. Does **not** relax `destructive_action_gate`/`git_safety` (force-push, `reset --hard`, discarding uncommitted work still require confirmation) — those are a different risk category. Still ask when genuinely blocked (missing info with no reasonable default, or an irreversibly-the-user's-call decision like which task to pick up next) — narrow exception, not the default.

- 2026-07-20: `AskUserQuestion` tool failed twice this session with a tool-permission-stream error before a response was received (once for the TASK-007 Qibla folder-depth question, once for the TASK-015/016 Dua/Azkar scope + favorites-storage question). Rather than block, proceeded with the best-reasoned option each time and documented the decision for Dipu to override: TASK-007 used the full repository-pattern folder split; TASK-015 marked done as verify-only (both features already existed pre-migration, rebuilding would have duplicated working code); TASK-016's Azkar favorites used SQLite (matching Dua's existing favorites pattern in `DatabaseService`) rather than Hive/hive_ce, since the hive staleness question below was still unresolved and SQLite avoided adding a new persistence dependency for one feature. This is a process note, not a locked technical decision — the SQLite-vs-hive_ce question below is still open.

- 2026-07-21: `AskUserQuestion` was explicitly rejected by Dipu when offered for the Epic 8 scope questions (home-screen widget scope, Mosque Finder, Hajj guide source), followed immediately by "continue task one by one don't stop" — read as a reaffirmation of the 2026-07-19 no-confirmation-gate instruction, not a one-off override. Proceeded with best-reasoned defaults on all three (see below and `AMAR_DEEN_PLAN.md`/`PROGRESS.md` Epic 8 entries), each explicitly flagged as reversible pending Dipu's actual input, same pattern as the 2026-07-20 `AskUserQuestion` failures above.
- 2026-07-21: Mosque Finder decided **out of scope by default** for this build (see the "Open questions" entry below, now resolved). No existing code for it in the repo. A real implementation needs a mosque-location POI dataset, either bundled (no legitimate offline dataset available/sourceable in this environment) or live-fetched (breaks the offline-first non-negotiable, needs a paid/rate-limited maps API this project has no credentials for). Reversible the moment Dipu supplies a dataset or an API budget — this is a default-to-safe placeholder decision, not a permanent architectural lock.
- 2026-07-21: Home-screen widgets (TASK-031) scoped **Android-only for v1** (`home_widget` package), iOS WidgetKit explicitly deferred — a Swift Xcode extension target can't be scaffolded without Xcode available, unlike Android's Kotlin/XML boilerplate which at least follows a well-known standard shape. Widget content scoped to next-prayer name + countdown, reusing `TimingBloc`'s already-computed data (no new domain logic). Full native implementation plan lives in `PROGRESS.md`; not implemented this session — needs a real Gradle/Kotlin toolchain to verify safely, unlike every other task this session.
